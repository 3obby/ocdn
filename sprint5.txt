═══ YOUR TASK: SPRINT 5 — PORTAL API ═══

build the Next.js API routes that serve data from the indexer database and accept write requests that feed into the tx construction pipeline. these are the endpoints the frontend will call.

the tasks from tasks.txt:
  [ ] Next.js API routes or separate service
  [ ] GET /api/feed — grouped by topic, ranked by burns, paginated
  [ ] GET /api/feed?sort=new|top — flat feed, sorted
  [ ] GET /api/topic/:hash — posts under topic, ranked
  [ ] GET /api/post/:hash — single post with full metadata
  [ ] GET /api/thread/:hash — full ancestor/descendant chain with depths
  [ ] GET /api/search?q= — full-text search over indexed content
  [ ] GET /api/author/:pubkey — post history, burn totals, key age
  [ ] GET /api/info/:hash — complete metadata (content_hash, block, tx, confirmations, burns)
  [ ] POST /api/post — submit text + topic, portal constructs + broadcasts
  [ ] POST /api/reply — submit text + parent_hash, portal constructs + broadcasts
  [ ] POST /api/burn — submit target_hash + amount, portal constructs + broadcasts
  [ ] POST /api/signal — submit refs list, portal constructs + broadcasts
  [ ] cost estimation endpoint: return expected sats for a given action at current feerate
  [ ] SSE /api/events — real-time feed updates on new blocks/content
  [ ] rate limiting + abuse prevention

use Next.js App Router route handlers (src/app/api/.../route.ts).

important design notes:
  - READ endpoints query the Prisma DB populated by the indexer. import { prisma } from "@/lib/db".
  - WRITE endpoints use the tx construction pipeline from src/lib/bitcoin/tx.ts and wallet.ts.
  - the write flow is: validate input → create protocol envelope (src/lib/protocol/create.ts) → reserve UTXO → build tx → broadcast → record PendingTx → return content_hash + status to client.
  - for POST/REPLY: the portal generates a keypair for the author (or accepts a client-provided pubkey in the future). for MVP, the portal is the author — it signs with its own key.
  - for BURN: the portal constructs the OP_RETURN tx targeting the given content_hash or topic_hash.
  - for SIGNAL: accepts an array of refs, constructs the OP_RETURN.
  - the cost estimation endpoint should return the expected sats for each action type at the current feerate (uses estimateCommitRevealCost, estimateBurnCost, estimateSignalCost from tx.ts plus the portal rake).
  - pagination: use cursor-based pagination (last contentHash or id) for feeds and topic views.
  - the /api/thread/:hash endpoint needs to walk the ThreadTree relation (ancestors + descendants) and return items with depth numbers, matching what buildThread() in mock-data.ts currently does.
  - the /api/feed grouped mode should match getGroupedFeed() behavior: group by topic, show top N root posts per topic ranked by burns, topics ranked by totalBurned.
  - for /api/search, use PostgreSQL full-text search (to_tsvector/to_tsquery) or ILIKE as a starting point. a prisma raw query is fine.
  - /api/info/:hash should return the full on-chain metadata: content_hash, txid, block hash, block height, confirmations (current tip - block height + 1), total burns, author pubkey, topic.
  - SSE /api/events: use a ReadableStream that polls the DB for new content since the client's last-seen height. keep it simple — poll every few seconds, send events when new content appears.
  - rate limiting: simple in-memory rate limiter (IP-based, e.g. 10 writes/minute, 60 reads/minute). doesn't need to be Redis-backed for MVP.

BigInt handling: Prisma returns BigInt for amount/fee/totalBurned fields. JSON.stringify cannot serialize BigInt. convert to number (if safe) or string in API responses.

match the response shapes to what the frontend already expects from mock-data.ts where possible — the types are Post, Topic, ThreadItem, TopicGroup defined there.

to test, we use 'npx next build'; no need to run the dev server. the prisma database has valuable live data, please append-only to .env.
