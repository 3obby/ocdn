═══ YOUR TASK: SPRINT 4 — INDEXER ═══

build the block indexer. this is the core infrastructure that scans the Bitcoin blockchain for OCDN protocol transactions and populates the database. it will run as a long-lived process on the VPS alongside bitcoind.

the tasks from tasks.txt:
  [ ] block scanner: iterate blocks from genesis (or start height)
  [ ] witness envelope detector: scan Taproot script-path spends for "ocdn" tag
  [ ] multi-envelope support: parse ALL envelopes per witness, not just first
  [ ] OP_RETURN scanner: detect "ocdn"-prefixed outputs
  [ ] POST indexing: parse envelope, verify sig, store content + topic_hash
  [ ] REPLY indexing: parse envelope, verify sig, validate parent exists, store
  [ ] BURN indexing: parse OP_RETURN, compute tx fee (input values - output values), record
  [ ] SIGNAL indexing: parse ref list, extract signer pubkey, compute fee, record
  [ ] topic normalization: lowercase + trim + NFC for canonical topic hashes
  [ ] thread tree builder: parent_hash chains → nested structure
  [ ] ranking computation: total sats burned per content_hash and topic_hash
  [ ] reorg handler: chain tip cursor, rewind on fork, re-process new chain
  [ ] real-time mode: subscribe to new blocks, index incrementally
  [ ] backfill mode: full historical scan
  [ ] external protocol detection: recognize EternityWall and other known inscription formats
  [ ] confirmation tracking: 0-conf display, 1-conf indicator, 6+ settled

  (note: the PostgreSQL schema task is already done — it's in prisma/schema.prisma)

key details from vision.txt you need:

  INDEXER RULES (from the spec):
    for every new block:
    1. witness scan — for each tx with a Taproot script-path spend, inspect witness stack for OP_FALSE OP_IF OP_PUSH "ocdn" ... parse envelope fields. compute content_hash from (pubkey, nonce, parent_ref, content). verify author signature. index content. for POST with topic, index under topic_hash.
    2. OP_RETURN scan — for each tx output that is OP_RETURN starting with "ocdn", parse version+type. BURN (0x03): extract 32-byte target_hash, compute tx fee (sum inputs - sum outputs), record as burn. SIGNAL (0x06): parse ref list, extract signer pubkey from tx input witness, compute fee, record.
    3. reorg handling — maintain chain tip cursor (block hash + height). on reorg: rewind index to fork point, delete all data from orphaned blocks (CASCADE delete via IndexedBlock relations), re-process new chain.

  BURN FEE COMPUTATION:
    burn_amount = tx_fee = sum(input values) - sum(output values)
    at RPC verbosity 2+, input values are available via prevout field on vin entries
    the OP_RETURN output has 0 sats value

  SIGNAL IDENTITY:
    in self-signed mode, the tx input signer IS the author. extract the signer pubkey from the first input's witness data (it's a Taproot key-path spend, pubkey derivable from the witness).

  TOPIC NORMALIZATION:
    the indexer normalizes text (lowercase, trim, NFC Unicode normalization) to compute canonical topic hashes for grouping. the on-chain data is the verbatim string.

  MULTI-ENVELOPE:
    the indexer must scan for ALL envelopes in a witness, not just the first. the multi-envelope-per-script pattern supports batching.

architecture:
  - put the indexer code in src/lib/indexer/ (scanner.ts, processor.ts, reorg.ts, etc.)
  - the main entry point should be a runnable script (e.g. src/indexer.ts or scripts/indexer.ts) that can be executed with `npx tsx scripts/indexer.ts`
  - use the existing BitcoinRpc client from src/lib/bitcoin/rpc.ts for block fetching
  - use the existing protocol parsers from src/lib/protocol/ for envelope parsing and verification
  - use the existing Prisma client from src/lib/db.ts for database writes
  - use getBlock(hash, 2) for verbosity-2 block data (includes prevout values on inputs, needed for fee calculation)
  - for backfill: iterate blocks sequentially, batch DB writes per block
  - for real-time: poll getBestBlockHash() every ~10 seconds (or ZMQ if you want), process new blocks
  - update TopicAggregate and AuthorAggregate tables as content is indexed
  - all DB writes for a single block should be in one transaction (atomic per block)

write tests for the envelope detection and fee computation logic.
