important design notes:

  DATA FETCHING PATTERN:
    - this is a "use client" app (page.tsx has "use client" at top)
    - use React state + useEffect for fetching, or a simple useFetch/useSWR hook
    - keep it simple: fetch() in useEffect, loading/error states, no complex state management
    - for infinite scroll: track nextCursor, fetch more on scroll-to-bottom

  TOPIC STRIP:
    - currently uses hardcoded TOPICS array
    - replace with a fetch to GET /api/feed?sort=topics (the grouped response includes all active topics)
    - or add a dedicated GET /api/topics endpoint that returns TopicAggregate[] sorted by totalBurned
    - the topic strip should show real topics from the indexed data
    - topics with no name should show as truncated hash (already handled by topicLabel())

  FEED:
    - "topics" mode: call GET /api/feed?sort=topics, render TopicGroup[] exactly as now
    - "new" mode: call GET /api/feed?sort=new, render flat Post[]
    - "top" mode: call GET /api/feed?sort=top, render flat Post[]
    - when topicFilter is set: pass &topic=<hash> to the API call
    - cursor pagination: track nextCursor from response, load more on scroll

  THREAD VIEW:
    - call GET /api/thread/:hash
    - response is { thread: ThreadItem[] } where ThreadItem = Post & { depth: number }
    - the existing ThreadView component already renders ThreadItem[] with depth-based indentation
    - just replace buildThread(postId) with the API fetch

  SEARCH:
    - call GET /api/search?q=<query>
    - debounce input (300ms) to avoid hammering the API
    - show loading state while fetching

  COMPOSE:
    - POST /api/post { content, topic } for new posts
    - POST /api/reply { content, parentHash } for replies
    - show the cost estimate (from GET /api/costs) in the compose sheet
    - on submit: show "broadcasting..." → "awaiting confirmation" → success/error
    - for MVP (pre-payment gate): posts go through immediately

  PROTOCOL BADGE:
    - Post type now has `protocol: string` ("ocdn" or "ew")
    - for EW posts, show a small "EW" badge or indicator in the FeedCard
    - EW posts have no nonce/signature (they're external inscriptions from Eternity Wall)
    - they can still receive burns and appear in search

  LOADING & ERROR STATES:
    - show a minimal loading indicator (pulsing "—" or similar, matching the minimal aesthetic)
    - on API error: show a subtle error message, allow retry
    - on empty results: show "—" (already handled for search)

  KEEP mock-data.ts:
    - keep the type definitions (Post, Topic, ThreadItem, TopicGroup, SortMode)
    - keep the helper functions (formatSats, formatTime, shortPubkey, topicLabel)
    - remove or keep the hardcoded data (it won't be imported anymore)

  REAL-TIME UPDATES (SSE):
    - connect to GET /api/events?since=<currentTipHeight>
    - on "posts" event: prepend new posts to the feed
    - on "burns" event: update burnTotal on affected posts
    - on "tip" event: update confirmations on visible posts
    - reconnect on disconnect with exponential backoff
