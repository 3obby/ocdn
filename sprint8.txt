SPRINT 8: EPHEMERAL LAYER — NOSTR + PROOF-OF-WORK
==================================================

Current state: sprints 1-7 done. Bitcoin durable posts work end-to-end.
Payment gate (sprint 6) handles the paid path. Frontend wired to real data
(sprint 7). No free posting tier exists.

═══ THE NEW MODEL ═══

Every compose action produces an ephemeral post by default. No wallet,
no sats, no account required. Ephemeral posts live on Nostr relays and
in the portal cache with a TTL. A CTA on each ephemeral post offers two
upgrade actions:

  1. BOOST (PoW) — browser mines NIP-13 proof-of-work in a WebWorker.
     User can run this for any duration. Higher difficulty = more weight =
     longer cache TTL + higher ephemeral ranking. Free, costs only compute.

  2. MAKE PERMANENT — pay sats → Bitcoin inscription → durable forever.
     Follows the existing Sprint 6 payment gate flow.

Bitcoin-layer posts (durable) are the canonical anchors. Nostr-layer posts
(ephemeral) are the living debate attached to them. A post can begin
ephemeral and be promoted to durable at any time by anyone willing to pay.

Two ranking dimensions coexist on every post:
  - economic weight: total sats burned (Bitcoin layer, existing)
  - compute weight:  sum of 2^difficulty across all PoW reactions (Nostr layer)

═══ EPHEMERAL VISIBILITY RULES ═══

The home/default feed is a paid-content-only surface. Ephemeral posts from
other users never appear in it. The rules are:

  Home feed (default — collapsed topic cards):
    - Shows durable Bitcoin posts only.
    - Exception: YOUR OWN ephemeral posts appear immediately after you post,
      optimistically inserted at the correct position. No one else sees them
      here. They disappear from this view if you reload and the session key
      is gone (true ephemeral UX — no promises).
    - Ephemeral count badges ("💬 12") are shown on durable FeedCards to
      signal that discussion exists inside — but the discussion is never
      surfaced unprompted in the home feed.

  Placement of YOUR ephemeral post (optimistic, immediate):
    - No topic selected → injected at the top of the "new" sort feed,
      rendered as EphemeralPostCard above the durable content.
    - Topic selected → injected inline at the top of that topic's collapsed
      group, as if it were a child of the topic anchor. Does not push durable
      posts down — it appears inside the topic's thread context.
    - Reply to a specific post → injected at the top of that post's ephemeral
      section within ThreadView, immediately visible when the thread is open.

  Thread view (expanded — user opened a specific post):
    - Durable Bitcoin reply chain renders as before.
    - Below it: a collapsed "N ephemeral replies" section, expandable on tap.
    - Inside that section: ALL ephemeral replies (yours and others), sorted
      by upvoteWeight desc (highest PoW first). Others' posts are visible
      here because the user has opted into the thread context.
    - Your own new reply appears optimistically at top of this section
      immediately on post, before server confirmation.

  Topic view (user tapped a topic chip — filtered topic feed):
    - Shows durable posts for that topic.
    - A "show ephemeral" toggle at the top of the topic feed (off by default).
    - When toggled on: ephemeral posts for that topic appear interleaved,
      styled as EphemeralPostCards. The toggle state persists per-session.
    - Your own ephemeral posts for this topic always show regardless of toggle.

  Profile view (user has a linked Bitcoin identity — see below):
    - Shows all durable posts by that pubkey.
    - Shows all ephemeral posts by the linked Nostr pubkey(s) for this session.
    - No visibility restrictions — it is your own history view.

  Summary table:
    Context               | Your ephemeral | Others' ephemeral
    ─────────────────────────────────────────────────────────────
    Home feed             | Yes (optimistic) | No (num indicator only)
    Topic feed            | Yes (optimistic) | (expandable bottom/replies only)
    Thread view           | Yes (optimistic) | Yes (collapsed, expandable)
    Profile view          | Yes              | No

═══ SESSION IDENTITY & PROFILE ═══

When a user completes a payment (Sprint 6 flow), they have demonstrated a
Bitcoin identity (the portal signed a tx attributable to a pubkey). At that
moment, associate their current session's Nostr pubkey with that Bitcoin
pubkey and persist it in localStorage.

  localStorage key: "ocdn_identity"
  Value: { bitcoinPubkey: string, nostrPubkeys: string[], linkedAt: number }
  nostrPubkeys is an array — each new session that completes a payment adds
  its Nostr pubkey to the list (capped at last 10 sessions).

On app load:
  - Check localStorage for "ocdn_identity"
  - If found: show a profile icon in the nav (bottom nav or top bar)
  - Profile icon tap → opens profile view filtered to bitcoinPubkey
  - Profile view shows: all durable posts by this pubkey, all ephemeral posts
    from any linked nostrPubkey, total burns given/received, pubkey (shortened)

Sprint 6 retrofit (new task in sprint 8):
  - In the payment completion handler (ComposeSheet or wherever sprint 6
    calls the success callback), call linkSessionIdentity(bitcoinPubkey)
  - linkSessionIdentity(bitcoinPubkey): reads current session Nostr pubkey
    from sessionStorage ("ocdn_nostr_priv"), derives its pubkey, appends to
    localStorage "ocdn_identity" record
  - This is a client-only operation — no new API route needed

New component: ProfileIcon (src/components/profile-icon.tsx)
  - Renders in BottomNav (or top bar) only if localStorage has ocdn_identity
  - Shows a small avatar-like icon: first 4 chars of bitcoinPubkey as monogram
  - Tap → opens profile sheet (full-screen slide-up, same pattern as ThreadView)
  - Profile sheet: "your posts" feed (GET /api/author/[bitcoinPubkey]),
    ephemeral posts list from linked nostrPubkeys, stats row

═══ NOSTR PROTOCOL CONVENTIONS ═══

All OCDN Nostr events use the following tag conventions:

  Post / reply (kind 1):
    ["t", "ocdn"]                              — required, for relay filter
    ["t", "<topic>"]                           — optional, topic name
    ["ocdn-ref", "<content-hash>"]             — Bitcoin-layer parent hash (if replying to durable post)
    ["e", "<parent-nostr-id>", "", "reply"]    — NIP-10 thread reference (if replying to ephemeral)
    ["nonce", "<nonce>", "<difficulty>"]       — NIP-13 PoW (required, minimum difficulty 8)

  PoW upvote / boost (kind 7, NIP-25):
    ["e", "<target-nostr-event-id>"]           — target ephemeral post
    ["ocdn-ref", "<content-hash>"]             — or target Bitcoin-layer post
    ["nonce", "<nonce>", "<difficulty>"]       — NIP-13 PoW (minimum difficulty 12 to count)

Relay list (configurable via NOSTR_RELAYS env):
  wss://relay.damus.io
  wss://relay.nostr.band
  wss://nos.lol
  wss://relay.primal.net

═══ DATA MODEL ═══

New Prisma models to add to prisma/schema.prisma:

  EphemeralPost {
    nostrEventId     String    @id                 -- Nostr event ID (hex, 32 bytes)
    nostrPubkey      String                        -- author's Nostr pubkey (ephemeral per-session)
    content          String
    topic            String?
    topicHash        String?
    parentContentHash String?                      -- references Post.contentHash (if child of durable)
    parentNostrId    String?                       -- references EphemeralPost.nostrEventId (if nested)
    powDifficulty    Int       @default(0)         -- difficulty achieved by author at post time
    upvoteWeight     BigInt    @default(0)         -- sum of 2^difficulty across all boosts received
    rawEvent         Json                          -- full signed Nostr event object
    expiresAt        DateTime                      -- createdAt + TTL (see TTL rules below)
    promotedToHash   String?                       -- contentHash if promoted to Bitcoin layer
    createdAt        DateTime  @default(now())

    parent           Post?     @relation(fields: [parentContentHash], references: [contentHash])
    children         EphemeralPost[] @relation("EphemeralThread")
    parentEphemeral  EphemeralPost?  @relation("EphemeralThread", fields: [parentNostrId], references: [nostrEventId])
  }

  NostrBoost {
    id               Int       @id @default(autoincrement())
    nostrEventId     String    @unique                -- the kind:7 reaction event ID
    nostrPubkey      String
    targetNostrId    String?                          -- ephemeral target
    targetContentHash String?                         -- Bitcoin-layer target
    powDifficulty    Int
    powWeight        BigInt                           -- 2^powDifficulty, precomputed
    rawEvent         Json
    createdAt        DateTime  @default(now())
  }

  indexes: topicHash, parentContentHash, parentNostrId, expiresAt, targetNostrId, targetContentHash

TTL rules (expiresAt):
  base TTL:                      24 hours
  per difficulty point above 8:  +1 hour (difficulty 20 → base + 12h = 36h)
  per boost received (difficulty ≥ 12): +30 minutes, capped at 7 days total
  promoted posts:                TTL cleared (never pruned, kept as shadow record)

═══ API ROUTES ═══

New endpoints:

  POST /api/ephemeral/publish
    Body: { signedEvent: NostrEvent }  — client-built, signed, ready to broadcast
    Actions:
      1. Validate Nostr event signature (verify schnorr, check id hash)
      2. Check minimum PoW difficulty (≥ 8 bits required to post)
      3. Store in EphemeralPost table
      4. Relay to configured Nostr relays (server broadcasts on behalf of client)
      5. Return { nostrEventId, expiresAt, powDifficulty }
    Rate limit: 30/min per IP

  POST /api/ephemeral/boost
    Body: { signedEvent: NostrEvent }  — kind:7 reaction event with PoW
    Actions:
      1. Validate event signature
      2. Require minimum difficulty ≥ 12 bits (prevents trivial spam)
      3. Store in NostrBoost table
      4. Update EphemeralPost.upvoteWeight (add 2^difficulty)
      5. Extend EphemeralPost.expiresAt per TTL rules
      6. Relay to Nostr
      7. Return { upvoteWeight, expiresAt }
    Rate limit: 60/min per IP (PoW is its own rate limiter)

  GET /api/ephemeral/[nostrEventId]
    Returns: { post: EphemeralPost, boosts: NostrBoost[], children: EphemeralPost[] }

  GET /api/ephemeral?parentHash=<contentHash>&topicHash=<hash>&sort=new|top&cursor=&limit=
    Returns ephemeral posts for a given parent or topic.
    parentHash: ephemeral children of a specific Bitcoin-layer post (for ThreadView)
    topicHash: ephemeral posts under a topic (for topic feed toggle)
    sort=new: by createdAt desc
    sort=top: by upvoteWeight desc
    Returns: { posts: EphemeralPost[], nextCursor }
    Note: this endpoint is never called for the home feed — home feed is durable-only.

  DELETE /api/ephemeral/[nostrEventId]  (internal, called by cleanup job)
    Prune expired posts. Only callable with API_WRITE_KEY.

Modify existing endpoints:

  GET /api/post/[hash]
    Add: ephemeralCount (count of EphemeralPost where parentContentHash = hash)

  GET /api/thread/[hash]
    Add: ephemeralReplies array on each ThreadItem
    (top N ephemeral children by upvoteWeight, not full recursive tree — keep it fast)

  GET /api/feed (all sorts)
    Add: ephemeralCount on each Post in response

═══ NOSTR CLIENT LIBRARY ═══

New file: src/lib/nostr/

  client.ts
    - generateEphemeralKeypair(): { privkey: Uint8Array, pubkey: string }
      generate a fresh secp256k1 keypair using @noble/curves. store privkey in
      sessionStorage (ephemeral per-tab, not persisted). this is the author identity
      for the session — NOT linked to any Bitcoin key.
    - buildPostEvent(content, topic?, parentContentHash?, parentNostrId?): UnsignedEvent
      construct kind:1 Nostr event with correct ocdn tags
    - buildBoostEvent(targetNostrId?, targetContentHash?): UnsignedEvent
      construct kind:7 Nostr event
    - signEvent(event: UnsignedEvent, privkey: Uint8Array): SignedEvent
      schnorr sign, compute event ID (sha256 of canonical JSON)
    - verifyEvent(event: SignedEvent): boolean
    - broadcastToRelays(event: SignedEvent, relays: string[]): Promise<BroadcastResult[]>
      open WebSocket connections to each relay, send EVENT message, await OK/NOTICE

  pow.worker.ts  (WebWorker — runs off main thread)
    Input message:  { eventTemplate: UnsignedEvent, targetDifficulty: number }
    Output messages: { type: "progress", currentDifficulty: number, nonce: number }
                     { type: "done", event: UnsignedEvent, difficulty: number, nonce: number }
    Algorithm:
      loop {
        set tags.nonce = [nonce.toString(), targetDifficulty.toString()]
        serialized = canonical_json(eventTemplate with nonce tag)
        id = sha256(serialized)
        bits = leading_zero_bits(id)
        if bits >= targetDifficulty → post "done" message, stop
        post "progress" every 10000 iterations
        nonce++
      }
    Note: use @noble/hashes/sha256 (already a dep). No WASM needed — JS SHA256
    is fast enough for difficulty ≤ 25 in-browser.

  pow.ts  (main thread interface to the worker)
    - startMining(event, targetDifficulty, onProgress): MiningHandle
      spawns pow.worker.ts, wires up message handlers
    - MiningHandle: { stop(): SignedEvent | null, currentDifficulty: number }
      calling stop() terminates the worker and returns the best event found so far
      (even if targetDifficulty not yet reached — partial PoW is fine)

  relay-sub.ts  (server-side, Node — for the portal/indexer)
    - subscribeToOcdnEvents(relays, onEvent): Subscription
      opens persistent WebSocket connections, sends REQ filter:
        { kinds: [1, 7], "#t": ["ocdn"], since: <last-seen-timestamp> }
      reconnects with exponential backoff on disconnect
      calls onEvent(signedEvent) for each incoming event
    - Used by the indexer process to ingest ephemeral content into EphemeralPost table

  index.ts  — barrel export

Dependencies to add:
  nostr-tools  (NIP parsing utilities, event ID computation, relay WebSocket handling)
  — use its EventTemplate / getEventHash / validateEvent / verifySignature
  — do NOT use its key generation (use @noble/curves already in project)
