OCDN — The Economic Speech Layer

PURPOSE: A permanent, economically-ranked knowledge layer where every post
is a Bitcoin transaction, every upvote is real sats, the entire social graph
lives on-chain, and the interface is a replaceable lens over indestructible
data. The protocol is Bitcoin. The product is the index. The website is a
commodity service that takes a flat rake.

---

CORE THESIS

1. Spam is a pricing problem. Bitcoin transaction fees filter infinite noise.
   The fee market itself is the quality signal — high-fee periods select for
   high-conviction content, low-fee periods open the floor.
2. Censorship resistance is maximized by putting data on Bitcoin. Suppression
   requires outbidding, policy capture, or sustained miner-level filtering —
   all expensive, all detectable, none silent.
3. Reading is free; writing costs sats. The cost of writing IS the spam
   filter AND the conviction signal. No separate mechanism needed.
4. The importance index is the product. Total sats committed to content
   and topics, weighted by author reputation and time. All inputs are
   on-chain and verifiable by anyone.
5. Everything is append-only. Nothing is edited or deleted — ever. Bitcoin
   is the append-only log. Edits are new transactions referencing originals.
6. The interface is a replaceable lens. Whoever controls the data controls
   the platform. Nobody controls Bitcoin. Competing websites index the same
   on-chain data. Corruption of any single interface has zero effect on
   data availability.
7. Identity is application-layer, anchored to Bitcoin keys. Author signatures
   live in the payload, not in Bitcoin's native UTXO signing model. Reputation
   is on-chain history — posts authored, citations received, burns given.
   Sybil cost = cost of building a real economic history over real time.
8. The entire social graph is on Bitcoin. Posts, replies, topics, burns,
   signals — all transactions. The graph is reconstructable from any
   archival node. No platform owns the graph.
9. Cheap signals compose into rich structure. Individual interactions are
   simple (an ordered list of plain-text references, ~80 sats). The
   assembly — many keys, many targets, over time — creates a navigable
   knowledge graph. Meaning lives in topology, not in any single signal.

INVARIANTS

Hard constraints. Cannot be designed around, only accommodated.

Bitcoin reality:

1.  Throughput is finite and contested.
    Block weight is capped. Every OCDN byte competes with all other Bitcoin
    demand for the same scarce blockspace. The system cannot exceed global
    throughput regardless of design.

2.  Inclusion requires competitive fee bidding.
    A valid transaction can still not be mined — fee too low, policy
    rejection, miner choice, targeted filtering. Liveness is a market,
    not a guarantee.

3.  Mempool policy is not consensus.
    Relay behavior, OP_RETURN limits, script acceptance, minimum feerate —
    all can change via policy flags without a fork. Only consensus rules are
    dependable for long-term liveness.

4.  Finality is probabilistic.
    Reorgs happen. "Published in block N" can be reversed for a window.
    Indexers must handle reorgs, conflicting tips, and ambiguous ordering
    during temporary forks.

5.  On-chain data is permanent, public, and irretractable.
    Every byte in witness or OP_RETURN is globally replicated by archival
    nodes and cannot be corrected. Errors, deanonymization, and regrettable
    content are forever.

6.  Archival is not universal.
    Pruned nodes discard old blocks. Durability means "as long as archival
    nodes exist," not "every node forever."

Architectural consequences:

7.  Authorship is application-layer.
    The Bitcoin tx signer is the UTXO spender. When the portal batches or
    pays fees, the portal is the signer. Author identity must be encoded
    in the payload and verified at the application layer.

8.  Batching requires payload-level identity.
    One tx, one txid, one set of input signers. Per-post authorship,
    reputation, and attribution live inside the batch payload, not in
    Bitcoin's native signature model.

9.  UTXO management is mandatory.
    Content-as-transaction requires: wallet state, UTXO selection, change
    management, fee estimation, RBF/CPFP, dust threshold compliance,
    mempool pinning awareness.

10. Burn semantics must be explicit.
    A high fee is just a high fee. A ranking burn requires an unambiguous,
    parseable on-chain marker tying excess fee to a content hash. Without
    protocol-defined structure, ordinary fees and burns are indistinguishable.

11. On-chain identity is a public preference graph.
    "Identity is a key" + "burns are on-chain" = full linkability of
    authorship and burn history. No author earnings to surveil, but
    burner preferences are public. Privacy is structurally compromised
    unless explicitly addressed.

12. Payload size is linearly coupled to fee.
    There is no discounted text lane. Nontrivial content increases cost
    proportionally. Witness discount (4x) helps but does not eliminate
    the coupling.

Infrastructure consequences:

13. Reconstruction requires infrastructure.
    Parsing blocks, detecting protocol events, maintaining a database,
    handling reorg rewinds, coping with missing historical data if the
    indexer is not archival. Non-optional for any portal.

14. The citation graph will be sparse.
    Fee-gated citations mean most content will have few or none. Graph
    algorithms over sparse data are noisy and gameable.

15. Tx construction is custodial or client-signed.
    If the portal builds transactions for users without them managing
    UTXOs, the portal controls funds (custodial) or requires client-side
    key management (wallet integration). No third option at Bitcoin's
    base layer.

16. Lightning receipt ≠ on-chain publication.
    A Lightning payment confirms payment, not inscription. The on-chain
    pipeline — UTXO management, fee policy, inclusion bidding — is a
    separate system with its own failure modes.

Ephemeral layer:

17. Parent must outlive children.
    An ephemeral post's expiration time must be ≥ the expiration time of
    every child (reply) beneath it. If a parent disappears while children
    remain, the children become orphans with no navigable context. This
    is enforced at write time: child expiresAt = min(computed_ttl,
    parent.expiresAt). Applies to both organic posts and seeded content.

TRANSACTION TYPES

Every transaction carries a protocol-defined payload in witness data or
OP_RETURN. The payload includes an author pubkey and an author signature
over the content — this is separate from the Bitcoin-level UTXO signature
and is what establishes authorship when the portal is the UTXO spender.

Post        Taproot tx, text in witness data (Taproot inscription
            style, ~1-5KB typical). Payload includes: author pubkey,
            author signature, optional topic string, content. A post
            with a topic competes under that topic — multiple posts
            to the same topic are siblings ranked by burns. A post
            without a topic is standalone. Topic identity is
            tagged_hash("ocdn/topic", topic_string). The tx fee is
            the minimum economic commitment.

Reply       Taproot tx referencing parent content hash. Text in witness.
            Threads are chains of Bitcoin transactions.

Burn        Tx with OP_RETURN referencing a 32-byte target hash +
            protocol marker. Target is a content hash (post-level
            burn) or topic hash (topic-level burn). The tx fee IS
            the burn — sats destroyed as ranking signal. Miners
            collect. Self-dealing costs face value (no refund path).
            The protocol marker disambiguates burns from ordinary
            transactions.

Signal      OP_RETURN tx carrying an ordered list of plain-text
            references — topics, entities, concepts as UTF-8 strings.
            Optional content_hash prefix for cross-referencing specific
            posts. References are human-readable on-chain; no hash
            lookup needed. The tx fee is the economic weight. Self-
            signed mode: user signs own tx, Bitcoin signature
            authenticates, no application-layer signature needed.
            ~80 sats at floor fees. Meaning lives in what you link
            and the order you link it. Single signals are simple;
            their assembly across keys and time creates a navigable
            knowledge graph.

Cross-ref   v2. Ordered list of 32-byte reference hashes in the
            Post/Reply envelope (ref_count + refs). No separate
            transaction type; references travel with the post.
            Each ref is a content_hash (cross-post) or topic_hash
            (cross-entity). Author-signed via refs_commitment in
            content_hash preimage. ~8 sats per ref at floor fees.

Edit        Tx referencing original content hash with EDIT relation.
            New content in witness. The website decides display (latest
            edit, full history, diff). Both versions permanent on-chain.

TEXT-ONLY PROTOCOL

The protocol handles only text. Images, video, documents are pointers —
text posts referencing external hashes. The protocol ranks the pointer;
the bytes live elsewhere (Blossom, IPFS, HTTP, etc.).

Text-only means:
- No blob storage problem. Text payloads are small relative to blockspace.
- Content moderation concerns largely dissolve (text, not binary).
- A million posts < 1 GB of block space.
- Archival nodes store the full history; pruned nodes do not (invariant 6).

ROLES

Portal        Clearnet interface. Runs an archival full node. Indexes all
(Website)     protocol-tagged transactions. Builds threading, topic
              grouping, ranking, search. Serves the UI. Takes a flat
              rake per write. Competes on UX and curation, not data
              control. May also display text from other Bitcoin
              protocols (e.g., EternityWall) alongside OCDN content.

              The portal is either custodial (holds UTXOs, constructs and
              signs txs on behalf of users) or requires client-side signing
              (wallet integration). In custodial mode, trust surface is:
              the portal could fail to publish after accepting payment.
              Mitigated by: proof of inclusion after mining, reputation,
              competing portals, and the option to self-publish.

Author        Submits content to the portal (or self-publishes directly).
              Signs the payload with their own key — authorship lives in
              the application-layer signature, not the UTXO spend.
              Authors do not earn from the protocol. Reputation is
              on-chain history: posts, citations received, burns given
              (invariant 11). The content's visibility IS the reward.

Burner        Destroys sats as ranking signal for content. The tx fee is
              the burn — miners collect, sats are irrecoverable. The act
              of burning IS the ranking signal. No intermediary decides
              what gets promoted. Self-dealing costs face value.

Reader        Free access. The website serves content ranked by economic
              commitment. Anyone can run their own indexer against the
              public chain (requires archival node + infrastructure).

WEBSITE ECONOMICS (THE FLAT RAKE)

The website provides four things:

1. Transaction construction + UTXO management — users don't run Bitcoin
   Core. Submit text + payment, the portal manages wallet state, fee
   estimation, RBF/CPFP, and broadcast. This is custodial by default
   (the portal holds the publishing UTXOs). Client-side signing via
   wallet integration is the non-custodial alternative.
2. Batching — amortize tx costs across many users. N posts in one Taproot
   tx via Merkle tree in witness data. Each post gets a Merkle proof of
   inclusion. Per-post authorship via payload-level signatures (invariant 8).
3. Indexing — scan the chain for protocol-tagged txs, build thread tree,
   compute rankings, handle reorg rewinds (invariant 4).
4. UX — search, threading, profiles, notifications, the forum interface.

Revenue model:
- Flat rake per post (e.g., 500–2000 sats on top of miner fee)
- Fiat on-ramp ("pay $1 to post" — portal handles all Bitcoin)
- Premium features (analytics, API access, notifications) as subscription

Cost math at 1 sat/vbyte (low-fee environment):
  Single post, 1KB:  ~1,250 sats miner fee + 1,000 rake = ~2,250 sats
  Batched, 100/tx:   ~12 sats miner fee + 1,000 rake    = ~1,012 sats
  Batched, 1000/tx:  ~1 sat miner fee + 1,000 rake       = ~1,001 sats

At scale, per-post cost approaches the rake itself. The website earns
margin on volume. Competition keeps the rake honest.

Note: these costs are for a low-fee environment. At 50 sat/vbyte a single
1KB post costs ~62,500 sats in miner fees alone. Fee volatility (invariant
2) means no fixed-cost UX promise is enforceable at protocol level. The
portal can absorb variance, queue during high-fee periods, or pass
through — each is a business decision, not a protocol guarantee.

Trust surface:
- The portal can censor its display (data still on Bitcoin, competitors
  can index it).
- The portal can fail to publish after accepting payment (detectable:
  no Merkle proof of inclusion; mitigated: reputation + competing portals).
- The portal can overcharge (users switch or self-publish).
- The portal can go down (data persists on-chain for anyone to re-index).
- The portal holds publishing UTXOs (custodial risk on the hot wallet,
  not on user content — content already mined is safe).

RANKING

Two ranking dimensions, both purely economic:

Topic ranking:    Total sats burned targeting topic_hash. Determines
                  which topics surface.

Post ranking:     Total sats burned targeting content_hash (including
                  the author's original tx fee). Determines which
                  posts surface within a topic or reply level.

All inputs are on-chain:
- Total sats committed = sum of all Burn tx fees referencing the
  target hash (content_hash or topic_hash).
- Author reputation = lifetime post history, burns received, key age.
  Publicly linkable (invariant 11).
- Time = block height of original post (subject to reorg, invariant 4).
  Two views: recency (decaying, for discovery) and canonical (cumulative,
  for reference). Both computable from the same on-chain data.

Signal ranking:   Signals contribute tx fees toward referenced topics
                  and content. Many cheap signals aggregate into
                  significant economic weight. Allocation of a signal's
                  fee across its multiple references is a portal-level
                  decision. The signal spending graph also provides
                  sybil-resistance inputs: graph topology, key diversity,
                  temporal patterns (see ADVERSARIAL ECONOMICS).

Anyone can compute the ranking from an archival node + indexer.
Competing websites can use different ranking algorithms over the same
data. The ranking is the product; the data is a public good. Rich
will be louder; poor will be unsuppressable.

BITCOIN-NATIVE PROPERTIES

Permanence          Archival nodes store the full history. Content durability
                    is a function of the archival node population, not of
                    any single operator. Stronger than any application-layer
                    storage guarantee, but not absolute (invariant 6).

Censorship          Suppression requires outbidding for blockspace inclusion,
resistance          miner-level tx filtering, or mempool policy capture.
                    All costly, all detectable, none silent. Not "impossible"
                    — maximally expensive relative to any known alternative.

Timestamping        Block inclusion IS the timestamp. Priority claims have
                    probabilistic finality — strong after 6+ confirmations,
                    effectively irreversible for practical purposes. Indexers
                    must handle the reorg window (invariant 4).

Identity            Application-layer author signatures over content, anchored
                    to Bitcoin keys. Portable across every portal that indexes
                    the chain. On-chain history accumulates as a public,
                    linkable financial record (invariant 11).

Ordering            Blocks provide total ordering after convergence.
                    During temporary forks, ordering is ambiguous (invariant
                    4). Indexers must handle conflicting tips.

Auditability        All protocol events are on-chain. Every sat committed to
                    every post is publicly verifiable from an archival node.
                    No custodial pools, no opaque systems — except the
                    portal's hot wallet for UTXO management.

ADVANCED CAPABILITIES (ALL BITCOIN-NATIVE)

Time-locked         OP_CHECKLOCKTIMEVERIFY / nLockTime. Commit content
content             hash now (proves existence), decryption key in a
                    time-locked tx (becomes spendable at block N).
                    Whistleblower protection, embargoed releases, dead
                    man's switches. No smart contracts, no oracles.

Multisig            M-of-N signatures to publish. Collective authorship
authorship          with cryptographic proof of consent. Editorial boards.
                    Whistleblower corroboration requiring N independent
                    co-signers. All on-chain, all verifiable.

Challenge           Bitcoin outputs locked by hashlock. "Pay N sats to
bounties            anyone who provides content contradicting claim X."
                    The bounty is a UTXO. The claim is the spend. No
                    intermediary adjudicates — the cryptographic
                    condition IS the adjudication.

Encrypted           Encrypt post to recipients' public keys. Ciphertext
content             on-chain (durable, timestamped, censorship-resistant),
                    only intended recipients decrypt. Private messaging
                    with the same durability guarantees as public posts.

Conditional         Taproot script conditions. Content revealed only when
revelation          conditions are met (hashlock, timelock, multisig).
                    Prediction markets as posts. Escrow-gated disclosures.

ADVERSARIAL ECONOMICS

Spam attack         Costs real Bitcoin per post. Thousands of spam posts =
                    millions of sats. Cluster of low-reputation keys spending
                    money for zero upvotes. Ranking ignores it automatically.

Censorship          Targeted: miner/pool filtering of specific protocol txs.
attack              Detectable (valid txs not being mined despite adequate
                    fee). Bypassable (different fee levels, different mempools,
                    direct miner submission). Sustained filtering requires
                    majority hashrate cooperation — expensive but not
                    impossible.

Erasure             Requires that all archival nodes discard the data.
attack              Practically very difficult — the Bitcoin archival node
                    population is large, geographically distributed, and
                    independently motivated. Not physically impossible, but
                    as close as any known system gets.

Reputation          Requires spending real sats over real time building
forgery             fake history. Cost scales linearly with the reputation
                    being forged. No shortcut.

Manipulation        Fake burns cost real sats — and those sats are
attack              destroyed. If you spend 1M sats manipulating
                    rankings, you've destroyed 1M sats for the privilege.
                    Sparse citation graph (invariant 14) may amplify
                    manipulation via graph-based metrics. Burner breadth
                    in the ranking function is the primary structural
                    defense — single-source burning is discountable.

Sybil               The signal spending graph is an unforgeable
detection            behavioral fingerprint per key. Temporal cadence,
(spending graph)     cross-content diversity, amount entropy, graph
                     topology, and reciprocity patterns distinguish
                     organic keys from sybil clusters. Every fake
                     graph edge costs real sats. Faking organic
                     topology requires sustained diverse spending
                     over months. Cost scales with time × diversity
                     × amount. The indexer need not censor — it
                     weights ranking by graph health metrics.

Defense cost is low (data replicated by existing Bitcoin infrastructure).
Attack cost is high and irreversible. The asymmetry is structural, not
policy-based — but not infinite. Every defense has a price at which it
breaks; the claim is that Bitcoin makes that price higher than any
alternative.

WHAT THE SYSTEM LOOKS LIKE

A forum where every post is a Bitcoin transaction. Upvotes are sats
burned — destroyed as ranking signal, collected by miners, irrecoverable.
The most burned-for content rises to the top. Nothing can be silently
deleted — everything is on archival nodes, append-only. Hateful noise is
low-ranked (nobody burns for it). Spam is priced out (every post costs real money).
Important content is permanently inscribed and economically endorsed.

The website is a window into on-chain data. If it disappears, someone
builds another one from an archival node — same data, same content.
The data is a public good. The lens is a competitive market. Bitcoin is
the settlement layer, the evidence archive, and the storage medium.

Authors do not earn. The content publicity
IS the value. No ads, no subscriptions, no editorial gatekeepers. The
accumulated economic conviction of every participant, weighted by their
verified on-chain history, determines what matters.

---

STRUCTURAL INCORRUPTIBILITY

Every content platform follows the same arc: launch with free speech
promises, grow, accumulate power over the data, get pressured by
governments/advertisers/ideology, gradually corrupt the editorial function.
The corruption is inevitable because the architecture permits it — whoever
controls the data controls the platform.

The pure Bitcoin model breaks this at the structural level. There is no
data controller to corrupt. The data layer is Bitcoin — no operator, no
board, no terms of service. The interface layer has operators, but they
are commodity services competing on UX over public data. Corruption of
any single interface has zero effect on data availability or integrity.

This is not "we promise not to censor." It is "censorship at the data
layer requires attacking Bitcoin itself, and censorship at the interface
layer is trivially circumventable." The strongest achievable property
given the invariants above.

DESIGN CHOICES

Why no relay layer    Bitcoin already solved persistence (for archival nodes).
                      Designing a relay incentive system (pools, bounties,
                      HTLCs) rebuilds a worse version of what Bitcoin provides
                      natively — one that introduces custodial pools, bounty
                      game theory, and relay sybil attacks.

Why no ephemeral      Every post costs real sats. This is a feature. The
tier                  absence of free posting IS the signal. Casual discourse
                      lives on existing platforms (Nostr, X, forums). OCDN
                      is the "this matters enough to spend money on" layer.
                      The cost is a costly signal (Spence, 1973) — it carries
                      information beyond the text. A post that costs sats is
                      inherently more credible than the same text posted for
                      free, because the author has something to lose.

Why no gradual        Content on Bitcoin is permanent. Ranking provides the
fade                  equivalent effect — underfunded content sinks in the
                      index. The bytes persist, but visibility is proportional
                      to economic commitment.

Why text-only         Text is small, cheap to inscribe, and fits witness data
                      efficiently. Media is pointers to external storage. The
                      protocol ranks the pointer; commodity storage markets
                      handle the bytes.

Why flat rake         Simple, sustainable, competition-driven. The website
                      earns by volume. No complex tokenomics, no custodial
                      pools, no fee curves. Users understand the cost upfront.
                      Competition drives it toward marginal cost.

Why custodial         Non-custodial tx construction requires wallet integration
portal default        on the client. Custodial is the pragmatic default for
                      web UX — the portal holds publishing UTXOs, not user
                      funds. Trust surface is bounded: worst case is
                      non-publication after payment, detectable and
                      reputation-costly. Non-custodial mode via client-side
                      signing is available for users who want it.

ENVELOPE FORMAT

OCDN data lives in Bitcoin witness via the same mechanism as Ordinals
inscriptions: an OP_FALSE OP_IF ... OP_ENDIF envelope inside a Taproot
script-path spend. This is not novel — it reuses an encoding with
established miner acceptance, relay propagation, and tooling. The
protocol defines what goes inside the envelope, not the envelope itself.

Encoding resilience: if OP_IF is restricted in Tapscripts (e.g.
BIP-110), an equivalent OP_DROP chain carries identical pushes:
  <pushN> OP_DROP ... <push1> OP_DROP <portal_key> OP_CHECKSIG
The push contents, ordering, and semantics are unchanged. The indexer
parses both patterns; the tx builder selects based on active consensus
rules. All fixed-size fields (tag, version, type, nonce, pubkey, sig,
hashes) are ≤64 bytes. Only content pushes need resplitting (520→256
byte max). Protocol-layer hashing and signatures are encoding-agnostic.

Bitcoin constraints on the envelope:

- Taproot script-path spends require a commit/reveal transaction pair.
  The commit tx creates a Taproot output whose script tree contains the
  envelope. The reveal tx spends it, placing the script (and data) in
  the witness. Two on-chain transactions per publication event. The
  indexer only parses reveal txs; commit txs are internal plumbing.

- Data pushes within the script are capped at 520 bytes. Larger payloads
  use multiple sequential pushes inside the envelope. Standard practice
  — same as Ordinals inscriptions.

- Witness data gets the 4x weight discount (1 weight unit per byte vs 4
  for non-witness). This is the economic basis for text-in-witness.

- Commit and reveal can be broadcast as a package. Bitcoin Core 28+
  supports 1-parent-1-child package relay. Both propagate together; the
  reveal confirms after the commit is mined.

The portal batches many posts into one commit/reveal pair. Per-post
overhead amortizes across the batch. The commit is the cost center
(creates the Taproot output); the reveal is the data carrier.

The envelope contains a fixed protocol tag that indexers scan for. Its
value is a spec-level decision. The tag must be short (minimize overhead),
unique (avoid collisions with Ordinals "ord" and other protocols), and
permanent (never changes — the version field handles evolution).

Note: OP_RETURN is the carrier for Burn transactions (38 bytes: tag +
version + type + content_hash). Content posts cannot fit in OP_RETURN —
per-post protocol overhead (author pubkey + signature + type/version)
exceeds 80 bytes before any content. The encoding split is resolved:
witness envelopes for content (Post, Reply), OP_RETURN for metadata
(Burn, and future Citation). This is two parsing paths, but each is
minimal and the split aligns with the data: content needs witness
space, burns need one transaction (no commit/reveal).

CONTENT ADDRESSING

Every protocol action references content by hash. Replies point to a
parent hash. Burns identify a target hash. Citations link source and
target hashes. Edits reference an original hash.

Required properties of the content hash:

  Deterministic.    Computable from the payload alone, before mining.
                    The author (or portal) can compute it at submission
                    time.

  Unique.           Distinct posts produce distinct hashes. Identical
                    text submitted twice by the same author must yield
                    different hashes. This implies a nonce or equivalent
                    uniqueness field in the preimage.

  Author-bound.     The same text from different authors produces
                    different hashes. The author's public key is part
                    of the preimage.

  Tx-independent.   Stable across batching, RBF replacement, and
                    rebroadcast. Cannot depend on txid (txid is unknown
                    at author signing time and shared across a batch).

SHA256 is the hash function. Bitcoin uses it. No reason to introduce
another.

The preimage includes at minimum: author pubkey, a nonce, and the
content bytes. Exact field ordering and serialization are spec-level
decisions. The constraint is the four properties above.

AUTHOR IDENTITY

Identity is a secp256k1 public key — specifically, a 32-byte x-only
public key as defined in BIP-340.

BIP-340 Schnorr is the signature scheme. Bitcoin-native (Taproot uses
it at consensus level), well-audited, production library support.
64-byte signatures.

The author signature is application-layer. It lives inside the envelope
payload, separate from the Bitcoin-level UTXO signature (which belongs
to the portal in custodial mode). The author signs the content; the
portal signs the transaction. Both use Schnorr, but over different
messages with different keys.

The signed message must commit to:
- The action type (prevents cross-type replay: a Post signature cannot
  be reused as a Citation).
- The nonce (same nonce as in the content hash).
- The content or reference fields (the substantive payload).

A fixed domain tag in the signed message prevents cross-protocol
signature reuse.

Key = identity. No key rotation mechanism in the initial protocol.
Rotation introduces complexity and an attack surface: a compromised
key could "rotate" the legitimate owner out before they react. An
author who loses a key starts fresh. This is Bitcoin's own model —
private key loss is final. Rotation is a future consideration, not
a launch requirement.

Reputation is the history of a key: posts authored, sats earned,
citations received, burns given, age of first appearance. All
derivable from the chain by any indexer. Sybil cost = cost of
building this history with real sats over real time.

PROTOCOL SPECIFICATION

Scope. Four transaction types: Post, Reply, Burn, Signal. Two
protocol versions: v1 (single-parent threading) and v2 (cross-
references). v1 is the minimum viable protocol — write text to
Bitcoin, organize by topic, thread replies, burn sats as ranking
signal, and link entities/concepts via cheap plain-text references.
v2 adds an ordered list of reference hashes to the Post/Reply
envelope, turning the thread tree into a navigable DAG. Each ref
is author-signed (committed transitively via content_hash), costs
~8 sats at floor fees, and is fully reconstructable by any indexer.
Edit and Batch are deferred to future versions. The type code space
accommodates them without breaking existing indexers.

Low-responsibility defaults (resolved):

  Encoding          Witness envelope for content (Post, Reply).
                    OP_RETURN for metadata (Burn). Content needs
                    witness for space; Burn needs one transaction,
                    not two. The split is pragmatic, not principled.

  Fee volatility    Pass-through. Portal charges rake + estimated
                    fee. Users bear fee-market risk. The portal
                    makes no promises about fee levels.

  Burn economics    Sats destroyed as miner fee. Ranking signal =
                    burn amount. Self-dealing costs face value (no
                    refund path). Authors do not earn from the protocol.

  Privacy           Public forum, public economics. No mitigations
                    in v1. Invariant 11 accepted.

  Reorg policy      Portal-level. Suggested: display after 1
                    confirmation with visual indicator; treat 6+
                    as settled.

  Cross-refs        v2 (see CROSS-REFERENCES below). v1 ranking
                    uses sats committed + time. v2 adds ref-based
                    DAG edges to the ranking surface.

  Policy risk       Acknowledged. v1 depends on current Taproot and
                    OP_RETURN relay policies. Witness envelope encoding
                    (OP_IF vs OP_DROP) is a pluggable layer; consensus
                    restrictions (e.g. BIP-110) require only an
                    alternative script structure, not protocol changes.
                    Burns and Signals (OP_RETURN ≤80 bytes) are immune.

  Bootstrap         Builders seed the ephemeral layer with external
                    content. Sources include 4chan /biz/, Reddit,
                    Hacker News, and similar high-signal forums.
                    Scraped threads become root ephemeral posts
                    (topics); scraped replies become child ephemeral
                    posts. Content is stripped to plain text, signed
                    by a dedicated seeder Nostr keypair, and ingested
                    via the /api/seed/ingest endpoint (API-key auth,
                    no PoW required — the key is the trust mechanism).
                    Seeded content expires under the same TTL rules
                    as organic posts; it serves as engagement bait
                    that real users can reply to, boost, or promote
                    to Bitcoin. The seeder pubkey is known; seeded
                    content is distinguishable but not visually
                    labeled (blends with organic content to avoid
                    a two-tier UX). Scraping runs daily via cron;
                    dedup is by deterministic Nostr event ID (same
                    input = same event = skip). No incentive mechanism
                    beyond the content itself — the seeder operates at
                    cost to bootstrap engagement.

PROTOCOL CONSTANTS

  Tag               "ocdn" (0x6F 0x63 0x64 0x6E)
  Version           0x01 (v1, no cross-refs)
                    0x02 (v2, cross-refs in envelope + content_hash)
  Type codes        POST = 0x01, REPLY = 0x02, BURN = 0x03,
                    SIGNAL = 0x06
                    Reserved: EDIT = 0x04, BATCH = 0x05

---

CONTENT HASH

  v1 (version = 0x01):

  content_hash = tagged_hash("ocdn/content",
      pubkey || nonce || parent_ref || content)

  v2 (version = 0x02):

  content_hash = tagged_hash("ocdn/content",
      pubkey || nonce || parent_ref || refs_commitment || content)

  refs_commitment:
      0x00 * 32                                  if ref_count = 0
      SHA256(ref_1 || ref_2 || ... || ref_n)     if ref_count > 0

  Ref order in the concatenation matches the envelope order. The
  commitment is order-dependent — the author chose the ordering and
  it is part of the signed statement.

  tagged_hash(tag, msg) = SHA256(SHA256(tag) || SHA256(tag) || msg)
  per BIP-340's tagged hash construction.

  pubkey:      32-byte x-only author public key
  nonce:       8 bytes, random (generated by author)
  parent_ref:  32 bytes — the post's position in the hierarchy:
               POST with topic:  tagged_hash("ocdn/topic", topic_string)
               POST standalone:  0x00 * 32
               REPLY:            parent content_hash
  content:     UTF-8 encoded text bytes
  ||:          concatenation

  The content hash is 32 bytes. It is the canonical identifier for a
  piece of content across all protocol actions (replies reference it,
  burns target it). Identical text under different topics or parents
  produces different content hashes. In v2, identical text with
  different cross-references also produces different content hashes —
  the references are part of what the author said.

  The version byte determines the preimage format. v1 and v2 content
  hashes for the same text are different (v2 preimage includes
  refs_commitment). The indexer must support both.

  Properties satisfied:
  - Deterministic (computable before mining)
  - Unique (random nonce)
  - Author-bound (pubkey in preimage)
  - Tx-independent (no txid dependency)
  - Position-bound (parent_ref in preimage)
  - Ref-bound (refs_commitment in preimage, v2 only)

---

AUTHOR SIGNATURE

  The signature commits to the content_hash, not the content fields.
  content_hash already binds (pubkey, nonce, parent_ref, content)
  in v1, and (pubkey, nonce, parent_ref, refs_commitment, content)
  in v2. The signature adds protocol context (version, type). No
  field is serialized twice. sign_msg is fixed-size and uniform
  across all action types.

  sign_msg = tagged_hash("ocdn/sign",
      version || type || content_hash)
  Preimage: 1 + 1 + 32 = 34 bytes, fixed.

  author_sig = BIP340_sign(author_privkey, sign_msg)

  Verification:
    BIP340_verify(author_pubkey, sign_msg, author_sig) → bool

  Serialization invariant: variable-length data (topic, content)
  appears in exactly one preimage each (topic_hash, content_hash),
  where it is terminal. sign_msg is always 34 bytes. All hierarchical
  context (topic, parent, cross-refs) is committed transitively via
  content_hash.

  Pubkey is committed transitively via content_hash and independently
  by BIP-340's internal challenge hash (R || P || m). Defense in depth.

---

WITNESS ENVELOPE (POST, REPLY)

  Follows the Ordinals envelope pattern: OP_FALSE OP_IF ... OP_ENDIF
  inside a Taproot script-path spend script.

  Envelope fields as sequential data pushes:

    "ocdn"              protocol tag, 4 bytes
    <version>           1 byte (0x01 or 0x02)
    <type>              1 byte (0x01 POST, 0x02 REPLY)
    <nonce>             8 bytes
    <pubkey>            32 bytes, x-only
    <sig>               64 bytes, BIP-340
    <topic>             variable UTF-8, POST only (empty push = standalone)
    <parent_hash>       32 bytes, REPLY only (omitted for POST)
    <ref_count>         1 byte, v2 only (0x00 if no refs; omitted in v1)
    <ref_1>             32 bytes, repeated ref_count times (v2 only)
    ...
    <ref_n>             32 bytes
    <content>           variable UTF-8, multiple pushes if >520 bytes

  The type byte determines the relational field: POST carries a topic
  push (variable, may be empty), REPLY carries parent_hash (fixed 32).
  Fixed fields first, then relational field, then refs (v2), then
  content.

  Each ref is a 32-byte hash — either a content_hash (cross-post
  link) or a topic_hash (cross-entity link). No per-ref type tag;
  the indexer resolves by lookup against known hashes. Collision
  between tagged_hash("ocdn/content", ...) and tagged_hash(
  "ocdn/topic", ...) is negligible (~2^-128). Unresolved refs are
  stored and resolved retroactively when the target is indexed.

  Ref semantics:
    Multi-parent reply    REPLY parent_hash = primary parent (tree).
                          refs = additional parents (DAG edges).
    Multi-entity post     POST topic = primary entity. refs contain
                          topic_hash values of additional entities.
                          One inscription, N entity associations.
    Evidence linking       refs = content_hashes of referenced posts.
                          "This claim cites that evidence."
    Entity merging        POST under topic A with ref to topic_hash(B).
                          Creates a cross-entity edge: A and B are
                          related. The portal decides display (merge
                          pages, "related entities" section, etc.).

  Overhead: v1 POST = 110 bytes + topic + content.
            v1 REPLY = 142 bytes + content.
            v2 POST = 111 bytes + topic + (32 * ref_count) + content.
            v2 REPLY = 143 bytes + (32 * ref_count) + content.
            No-ref v2 post adds 1 byte vs v1.

  The envelope lives in a Taproot script-path spend, requiring the
  commit/reveal transaction pair:

    Commit tx     Portal spends an existing UTXO. Creates a P2TR
                  output whose script tree contains the envelope
                  script. Change output back to portal.

    Reveal tx     Spends the commit output via script-path. The
                  witness contains the envelope script + Taproot
                  control block. Change output to portal.

  Content is on-chain when the reveal tx is mined. The commit and
  reveal can be broadcast as a package (Bitcoin Core 28+ 1P1C relay).

---

OP_RETURN MARKER (BURN)

  Single transaction, no commit/reveal. The tx fee IS the burn.

  Outputs:
    1. OP_RETURN:  "ocdn" || 0x01 || 0x03 || target_hash
                   (4 + 1 + 1 + 32 = 38 bytes, 0 sats)
    2. Change:     back to burner

  target_hash is a content hash (post burn) or topic hash (topic burn).
  No collision: different tagged_hash domains.

  burn_amount = tx_fee = sum(inputs) - sum(outputs)

  The indexer reads the OP_RETURN, extracts the target hash, checks
  it against known content hashes and topic hashes, computes the tx
  fee (requires input value lookup — available via getblock verbosity
  2+ with -txindex), and records the fee as economic commitment.

  No payment output. No author address. No UTXO bloat — the
  OP_RETURN is pruned from the UTXO set by consensus. Miners
  collect the burned sats, aligning their incentive to include
  high-burn txs (strengthens liveness, invariant 2).

  Self-dealing costs face value. To boost content by N sats, you
  destroy N sats. No refund path. Burner breadth in the ranking
  function is the additional structural defense (see RANKING).

---

OP_RETURN SIGNAL

  Single transaction, self-signed. The tx fee is the economic weight.

  OP_RETURN payload (up to 80 bytes):
    "ocdn"          4 bytes
    <version>       1 byte
    <type=0x06>     1 byte
    <refs>          ordered sequence of:
                      text ref:    [len][UTF-8 string]  (len > 0)
                      content ref: [0x00][8-byte content_hash prefix]

  74 bytes available for references after the 6-byte header.

  Examples:
    ["McDonald's"]["wage theft"]           24 bytes — links entity to concept
    ["Oracle"]["open source"][0x00][hash]   30 bytes — entity + concept + evidence
    ["Company A"]["worse than"]["Company B"] ~35 bytes — structured comparison

  Text references are human-readable on-chain. Any third party
  decodes them: length-prefixed UTF-8 strings after a fixed header.
  No lookup table, no chain replay needed, no bootstrapping problem.

  Content references (0x00 prefix) point to specific on-chain posts
  via 8-byte content_hash prefix. Resolvable by any indexer that has
  parsed the chain. Opaque in isolation, but honest about what they
  are.

  Identity: the tx input signer. In self-signed mode, the Bitcoin
  UTXO signature authenticates the signal — no application-layer
  signature needed. The signer's pubkey IS the author.

  In portal-facilitated mode: signals are batched into witness
  inscriptions with per-signal application-layer signatures.
  Amortized cost ~18-50 sats/signal. Custodial during batch window.

  The indexer normalizes text references (lowercase, trim, NFC
  Unicode normalization) to compute canonical topic hashes for
  grouping. The on-chain data is the verbatim string. Different
  portals may normalize differently — that is their curation
  value-add.

---

INDEXER RULES

  For every new block:

  1. Witness scan. For each transaction with a Taproot script-path
     spend, inspect the witness stack for a script containing:
     OP_FALSE OP_IF OP_PUSH "ocdn" ...
     If found: parse version byte.

     version 0x01 (v1): parse envelope fields in order. Compute
     parent_ref (topic_hash for POST, parent_hash for REPLY, null
     for standalone). Compute content_hash from (pubkey, nonce,
     parent_ref, content). Compute sign_msg from (version, type,
     content_hash). Verify author signature. Index the content.
     For POST with topic, index under topic_hash. For REPLY,
     validate that parent_hash references existing indexed content.

     version 0x02 (v2): parse envelope fields through the relational
     field (topic or parent_hash). Read ref_count (1 byte). Read
     ref_count * 32 bytes as refs. Remaining pushes = content.
     Compute refs_commitment: if ref_count = 0, use 0x00 * 32;
     else SHA256(ref_1 || ... || ref_n). Compute content_hash from
     (pubkey, nonce, parent_ref, refs_commitment, content). Compute
     sign_msg from (version, type, content_hash). Verify author
     signature. Index the content. For each ref, resolve against
     known content_hashes and topic_hashes. Index cross-post edges
     (ref matches content_hash) and cross-entity edges (ref matches
     topic_hash). Unresolved refs are stored and resolved retro-
     actively when the target is later indexed.

  2. OP_RETURN scan. For each transaction output that is OP_RETURN,
     check if payload begins with "ocdn". If so: parse version and
     type. If BURN (0x03), extract target hash (32 bytes). Check
     against known content hashes and topic hashes — no collision
     risk (different tagged_hash domains). Compute tx fee (sum of
     input values - sum of output values). Record the fee as burn
     amount for that content or topic.

     If SIGNAL (0x06), parse the reference list: iterate remaining
     bytes, reading length-prefixed fields. len > 0 → text reference
     (len bytes UTF-8). len = 0x00 → content reference (next 8 bytes
     as content_hash prefix). Extract signer pubkey from tx input
     witness (self-signed mode). Compute tx fee. Record signal as:
     (signer_pubkey, [refs], tx_fee, block_height). For each text
     ref, normalize and index under its canonical topic hash. For
     each content ref, resolve against known content_hash prefixes.

  3. Reorg handling. Maintain a chain tip cursor (block hash + height).
     On reorg: rewind index to the fork point, delete all data from
     orphaned blocks, re-process from the new chain. Content that
     appeared in an orphaned block and reappears in the new chain is
     re-indexed normally.

  Invalid envelopes (bad signature, unknown version with same tag,
  malformed fields) are silently ignored. The indexer processes only
  what it can fully validate. The indexer must support both v1 and
  v2 envelopes concurrently — version byte determines parsing path.

COST ESTIMATES (NO BATCHING)

  At 1 sat/vbyte (low-fee environment):

  POST, 500 bytes content, no refs (v1 or v2 ref_count=0):
    Commit tx:  ~150 vbytes                = ~150 sats
    Reveal tx:  ~250 vbytes (110 overhead
                + topic + 500 content,      = ~250 sats
                witness discount applied)
    Miner fees total:                        ~400 sats
    Portal rake:                            ~1,000 sats
    User cost:                              ~1,400 sats

  v2 ref cost delta (witness discount: 4x):
    ref_count byte:        +1 byte  = +0.25 weight units
    Per ref (32 bytes):    +32 bytes = +8 weight units
    1 ref:   ~8 extra sats at 1 sat/vB
    2 refs: ~16 extra sats
    5 refs: ~40 extra sats
    Multi-entity post linking 3 entities: ~24 sats above base cost.

  BURN:
    Single tx: ~80 vbytes (no payment       = ~80 sats relay minimum
    output, just OP_RETURN + change)
    Portal rake (if facilitated):            ~500 sats
    Burn amount:                         entirely user's choice
    Total cost:                   ~580 sats overhead + burn amount

  At 50 sat/vbyte (high-fee):
    POST becomes ~20,000 sats miner fee + rake = ~21,000 sats
    BURN relay minimum becomes ~4,000 sats + rake + burn amount
    Per ref becomes ~400 sats (still cheap relative to base cost)

  SIGNAL (self-signed OP_RETURN):
    Single tx: ~80 vbytes                = ~80 sats at 1 sat/vB
    No portal involvement
    User cost:                            ~80 sats minimum

  SIGNAL (portal-batched, witness inscription):
    Per signal amortized:                 ~18-50 sats at 1 sat/vB
    Portal rake per signal:               ~10 sats
    User cost:                            ~28-60 sats

  Note: POST cost estimates above are ~10% optimistic. Detailed
  calculation for a 445-byte payload gives commit ~155 vB + reveal
  ~271 vB = ~426 vB total. The estimates remain useful approximations.

  Batching (future) amortizes commit/reveal across N posts,
  approaching rake-only cost per post at scale.

CROSS-REFERENCES (v2)

  v2 envelopes carry an ordered list of 32-byte reference hashes
  between the relational field and content. These turn the single-
  parent thread tree into a directed acyclic graph (DAG).

  Design constraints satisfied:
  - Author-signed: refs are committed via refs_commitment in the
    content_hash preimage. The portal cannot inject or alter refs
    without breaking the signature.
  - Minimal cost: 1 byte (ref_count) + 32 bytes per ref, at witness
    discount. ~8 sats per ref at floor fees.
  - Fully reconstructable: any indexer scanning blocks with knowledge
    of the v2 envelope format reconstructs all DAG edges.
  - Backward compatible: v1 envelopes are still valid. The version
    byte determines the parsing path.

  Each ref is a raw 32-byte hash. No per-ref type tag — the indexer
  resolves by lookup against known content_hashes (cross-post edge)
  and topic_hashes (cross-entity edge). tagged_hash domain separation
  makes collision negligible (~2^-128). Unresolved refs (forward
  references, not-yet-indexed targets) are stored and resolved
  retroactively.

  Ref order is canonical — it matches the envelope encoding order
  and is preserved in refs_commitment. The author chose the order;
  the first ref is the most relevant.

  Signals (OP_RETURN, ~80 sats) remain the tool for standalone
  entity-to-entity association assertions that are not attached to
  a specific post. Refs are for post-bound associations: "this
  specific post references those specific things."

DEFERRED (FUTURE VERSIONS)

  Edit (0x04)         Witness envelope referencing original content
                      hash. Same author (indexer verifies pubkey).
                      Portal decides display: latest edit, full
                      history, diff view.

  Batch (0x05)        Multiple posts in one commit/reveal. Merkle tree
                      of content hashes in the envelope. Per-post
                      Merkle proof as inclusion receipt. Reduces cost
                      toward rake-only at volume.

  Key rotation        Signed attestation linking old key to new key.
                      Deferred due to attack surface concerns (see
                      AUTHOR IDENTITY).

  Privacy             Separate posting keys, batch-level unlinkability,
                      or other mitigations for invariant 11.

  Staking/equity      Content supporters earn dilutive stake; future
                      support distributes to existing stakeholders
                      with rounding bias toward author. Trustless
                      split enforcement requires covenant opcodes
                      (OP_CTV, OP_CAT). Trust-minimized variant
                      possible today via multi-output txs with
                      verifiable deterministic splits — patron
                      verifies before signing, anyone audits after
                      mining — but not trustlessly enforceable at
                      the address level.

OPEN QUESTIONS (REMAINING)

Most design questions are resolved above. What remains:

1. Commit/reveal pipeline. The two-transaction pattern is structurally
   required. How the portal manages this operationally — pre-funding,
   pipeline depth, failure recovery when a commit confirms but the
   reveal is stuck — is implementation-level but affects UX latency
   and reliability.

   Key-path recovery on all commit outputs. Non-negotiable. The portal's internal key is always the Taproot internal key. This is invisible to the protocol (the indexer only cares about the script-path reveal).
RBF signaling on all transactions. Fee-bumping is the primary tool for stuck transactions.
Package broadcast. Commit + reveal submitted together. Both can land in the same block.
Multi-envelope batching. Collect posts into a time window, multiple envelopes per script. No protocol change needed — the indexer already parses all envelopes it finds in a witness.
UTXO pool with fan-out. Pre-split for concurrency. Consolidate to avoid fragmentation. Background process.
Retry with escalation. Mempool eviction → rebroadcast. Stuck tx → RBF fee bump. Persistent failure → key-path recovery + refund.

the multi-envelope-per-script pattern should be explicitly blessed by the indexer rules — "scan for all envelopes in a witness, not just the first"

2. Content size limits. The protocol imposes none (fee market self-
   regulates). Should the portal impose a soft limit? Long content
   increases reveal tx size and cost linearly. Practical question,
   not protocol question.

   no limits imposed by portal, clear data displayed to the user.

3. Nonce generation. Spec says 8 random bytes. Who generates it —
   author (client-side) or portal? In custodial mode the portal can
   generate it. In non-custodial mode the author must. Affects the
   signing flow.

   author.

4. Signet/testnet validation. The spec should be validated against
   real Bitcoin transactions before mainnet. Signet is the appropriate
   testbed — realistic mempool behavior, free sats.

   sure, signet.

Here’s how the current “safe embed” tech maps to your case and how to use it.

---

## 1. **Sanitizer API** (HTML only)

For **4chan post text** (HTML with `<br>`, `<span class="quote">`, links, etc.):

- Use **`Element.setHTML(unsafe, { sanitizer })`** instead of `innerHTML`.
- The Sanitizer API strips scripts and dangerous markup; you can allow only safe tags/attributes.
- So you can **render post HTML safely** without XSS. This doesn’t cover images.

---

## 2. **Images: “transmit safely” = proxy + re-encode**

To **show images** “transparently” but **without giving users the raw third‑party bytes**:

- **Idea:** Never send the original file to the client. Fetch it on your side, **decode to pixels**, **re-encode to a new image**, then send or display that.
- Effects:
  - EXIF/metadata (and a lot of hidden tricks) are dropped.
  - You’re not serving 4chan’s exact bytes; you’re serving a new image you generated.
  - User only ever gets your (proxied and re-encoded) content.

**Ways to do it:**

| Where | How | Pros / cons |
|-------|-----|-------------|
| **Server** | Proxy the 4chan URL → decode (e.g. sharp) → re-encode (JPEG/PNG/WebP) → stream response | One place to enforce type/size; no client CPU; works in all browsers. |
| **Client** | Your proxy returns the image; JS does `fetch(proxyUrl) → createImageBitmap(blob) → draw to OffscreenCanvas → toBlob() → createObjectURL()` and you display the blob URL | No server image processing; re-encode happens in the browser so user still only sees “your” pixels, not raw 4chan file. |

So “new tech” that helps here is mainly **re-encoding** (server or client) plus **Sanitizer** for HTML. Both are widely supported.

---

## 3. **Fenced Frames** (and why they’re not the right tool)

- Fenced Frames are for **Privacy Sandbox** (ads, FLEDGE, Shared Storage).
- They **don’t** take a normal `src` URL; they need a `FencedFrameConfig` and special reporting/shared-storage setup.
- So they’re **not** a general “embed this image safely” mechanism for arbitrary 4chan media.

---

## 4. **Concrete “safe and transparent” setup**

- **HTML:** When you render 4chan post body, use the **Sanitizer API** (`setHTML`) so scripts and dangerous attributes are removed. Content is “transmitted” (shown) transparently, but only safe markup is rendered.
- **Images (and optionally video):**
  - **Proxy** the 4chan URL on your server (allowlist `i.4cdn.org`, validate `Content-Type`).
  - **Re-encode** before the user sees it:
    - **Server:** e.g. Next API route that fetches from 4chan, runs the bytes through **sharp** (or similar), outputs a new JPEG/WebP and streams it. User gets only your re-encoded image.
    - **Client:** Same proxy URL; in the page you `fetch` it, `createImageBitmap`, draw to canvas, `toBlob()`, then set an `<img src={objectURL}>` (or use the blob in a video element). User still only sees re-encoded pixels, not the original file.

That way you **show/transmit the content** (so it feels transparent), but you **don’t expose users to the raw resource**: no EXIF, no odd image tricks, and the browser never loads the original 4chan URL directly. The only “new” tech you need is Sanitizer for HTML and re-encoding (server or client) for media.
