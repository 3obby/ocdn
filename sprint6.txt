═══ YOUR TASK: SPRINT 6 — PAYMENTS ═══

integrate Lightning payments so that write actions (post, reply, burn, signal) require payment before the portal constructs and broadcasts the transaction. users pay the miner fee + a small portal rake via Lightning invoice.

the tasks from tasks.txt:
  [ ] BTCPayServer instance setup (or LND invoice generation directly)
  [ ] Lightning invoice generation per write action (post/reply/burn/signal)
  [ ] invoice amount = miner fee estimate + portal rake
  [ ] payment verification: invoice settled → queue tx for broadcast
  [ ] payment webhook: BTCPayServer → portal pipeline trigger
  [ ] fiat on-ramp: "pay $X to post" with BTCPayServer fiat conversion (stretch)
  [ ] payment status tracking: pending → paid → queued → broadcast → confirmed
  [ ] refund path: payment received but tx failed → credit or LN refund

important design notes:

  PAYMENT FLOW:
    1. client calls POST /api/post (or /reply, /burn, /signal) with the content
    2. portal validates input, estimates cost via estimateCommitRevealCost/estimateBurnCost/estimateSignalCost
    3. portal creates a Lightning invoice for (estimated fee + rake), stores the pending action
    4. returns { invoiceId, paymentRequest (bolt11), amountSats, contentHash (pre-computed) } to client
    5. client pays the invoice (via any Lightning wallet or in-app WebLN)
    6. on settlement: portal constructs the envelope, reserves UTXO, builds tx, broadcasts
    7. updates PendingTx status: paid → queued → revealed → confirmed
    8. client polls or receives SSE update with final status

  ARCHITECTURE OPTIONS (pick one):
    A. BTCPayServer — full-featured, handles LN node management, has webhook system, can do fiat pricing
       - deploy on VPS alongside Bitcoin Core
       - use BTCPayServer Greenfield API for invoice creation + webhook registration
       - webhook fires when invoice is settled
    B. Direct LND/CLN — lighter, no BTCPayServer overhead
       - connect to LND (already on VPS per creds.txt) or set up CLN
       - use lnrpc to create invoices and subscribe to invoice settlement stream
       - simpler but less batteries-included
    C. For signet MVP: mock payments — accept all posts for free, add payment gate later
       - fastest path to end-to-end testing
       - skip this sprint entirely for now, come back before mainnet

  THE PORTAL RAKE:
    - portal operator earns a small margin on each write action
    - suggested: 10-20% on top of the miner fee estimate
    - configurable via env var PORTAL_RAKE_PERCENT (default 15)
    - example: post costs ~1200 sats in miner fees → invoice for ~1380 sats

  SCHEMA ADDITIONS:
    - add to PendingTx or create new PaymentInvoice model:
      invoiceId, bolt11, amountSats, status (pending/paid/expired), paidAt, expiresAt
    - link PaymentInvoice → PendingTx so settlement triggers broadcast

  ENV VARS NEEDED:
    - BTCPAY_URL or LND_REST_HOST, LND_MACAROON_HEX, LND_TLS_CERT
    - PORTAL_RAKE_PERCENT

  CURRENT WRITE ENDPOINTS:
    the POST /api/post, /api/reply, /api/burn, /api/signal routes currently broadcast immediately.
    modify them to: validate → estimate cost → create invoice → return invoice to client.
    add a new endpoint or webhook handler that fires on payment settlement to do the actual broadcast.

  TESTING:
    on signet, Lightning payments work with signet LND. the VPS already has LND setup per creds.txt.
    alternatively, use a mock payment mode where all invoices auto-settle (for development).
